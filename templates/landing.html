The Problem Space

Five distinct failure modes.
One common root cause.

After five years running Keycloak at scale, the same breakdowns appeared repeatedly across every customer type — startup, enterprise, and everything in between. The tool is powerful. The surface area is immense. And the gap between "I need SSO" and "SSO is configured correctly" is enormous.

01
The UI has 7 tabs per client. Customers don't know which ones matter.

Keycloak's client configuration UI presents Settings, Keys, Credentials, Roles, Client Scopes, Mappers, and Advanced tabs simultaneously. For a non-security engineer trying to configure a React SPA for SSO, every field is a potential landmine. "Valid Redirect URIs", "Web Origins", "Root URL", "Admin URL" — similar names, radically different implications. A single trailing slash in the wrong field causes a redirect URI mismatch error that surfaces as a cryptic OAuth error in the browser.

"I spent 4 hours on a redirect URI mismatch. Turned out I had a trailing slash." — Operations engineer, customer onboarding call
02
Identity Provider configuration requires deep OIDC / SAML knowledge most customers don't have.

Connecting a corporate IdP — Azure AD, Okta, PingFederate — requires the operator to correctly supply Issuer URL, Authorization URL, Token URL, JWKS URL, Client ID, Client Secret, and then configure mappers for the claims the IdP actually returns. Each IdP names its claims differently. Azure uses upn where Okta uses email. Getting this wrong means users authenticate successfully at the IdP but arrive at the application with no name, no email, and no roles — a silent, baffling failure.

"Authentication worked. Login didn't. Spent 2 days figuring out the claim mapper was wrong." — Enterprise customer, SAML integration
03
Heavy Java automation covered the happy path. Edge cases still required manual intervention.

The team built sophisticated Java automation: realm creation scripts, default client templates, role hierarchies provisioned on signup. This handled 80% of setups. The remaining 20% — custom attribute mappers, non-standard SAML configurations, client-specific scope overrides — required a human being to click through the UI. That human was always an R&D engineer because operations didn't have the Keycloak depth to do it safely. The automation created a false confidence: "the standard setup works, so any deviation is someone else's problem."

"Our automation was great until a customer needed a custom SAML attribute mapper. Then it was a war." — R&D engineer
04
R&D engineers became the Keycloak support desk — at the cost of product velocity.

Every escalated Keycloak configuration issue pulled an R&D engineer out of the sprint. A typical incident looked like this: customer opens a ticket at 2pm, operations triages it and determines it needs Keycloak access, tickets R&D, R&D engineer picks it up after standup next morning, joins a Zoom call, screen-shares Keycloak UI, makes the change, closes the ticket. Total elapsed time: 18–24 hours. Total R&D time lost: 2–3 hours including context switch cost. Multiply by 400 customers. The math is brutal.

"I didn't ship a feature for two weeks during the Q4 enterprise onboarding wave. Just Keycloak." — Senior engineer, product team
05
Keycloak documentation is comprehensive. It is also 400 pages long and assumes prior knowledge.

The official Keycloak documentation is genuinely excellent — for someone who already understands OAuth 2.0, OIDC, SAML 2.0, and JWT internals. For an operations engineer onboarding their tenth enterprise customer of the quarter, "see RFC 7636 for the PKCE specification" is not helpful. The knowledge gap isn't about intelligence — it's about domain depth that takes years to accumulate. No operations team should need to read RFC 7636 to configure a client application.

"I know how to operate software. I don't know what a code_challenge_method is and I shouldn't have to." — Operations team lead

The Solution

KeycloakAI collapses the expertise gap.

The insight is simple: Keycloak's Admin REST API is well-defined and deterministic. The only hard part is knowing which endpoint to call, with what payload, in what order. That is exactly what an LLM is good at.

SSO-secured — only authorised users can act

Every session is authenticated through Keycloak itself via OIDC PKCE. The tool that manages Keycloak is protected by Keycloak. No credentials stored, no shared passwords.

Natural language in — structured API call out

Claude parses the English intent and extracts the exact JSON payload the Admin REST API expects. No UI. No tabs. No jargon. "Create a Google IdP with these credentials" becomes a correct API call in under a second.

Full transparency — every payload shown

Every operation displays the exact JSON that was sent and the HTTP response code received. Nothing is hidden. Operations engineers learn the API by using the tool — the tool teaches while it works.

Deterministic pipeline — no hallucinated endpoints

LangGraph routes every intent through a typed dispatch table. Claude identifies the intent; Python executes the correct pre-defined API call. The AI cannot invent endpoints or make up parameters.
